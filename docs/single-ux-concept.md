# Single UX Concept

## Product sentence

`Days verwandelt Gedanken, Signale und Ziele in einen klaren Tag.`

That sentence is the test for every future screen and feature.

## What `Single` should feel like

`Single` should feel:
- calm
- local
- private
- fast to read
- direct to correct

`Single` should not feel like:
- a collection of mini apps
- a workbench full of tools
- a self-rating dashboard
- a settings-heavy lab

## Home is the anchor

The current Home screen remains the visual anchor.

Rules:
- Home stays recognizable.
- Home is not a feed.
- Home is not a tool hub.
- Home is the primary work surface.
- The day track is the center.
- The day is shown in three broad blocks first.

## Start is a fixed tile surface

`Start` is no longer a mixed dashboard.

Rules:
- Start uses only large tiles.
- No button rows.
- No KPI strips outside tiles.
- No explainer sections outside tiles.
- Every future Start addition must be a tile in the same visual language.
- Iteration is allowed, but only by refining the fixed tile system, not by reintroducing mixed layouts.
- Start copy stays minimal.
- No interpretive coaching phrases on Start.
- No abstract status wording like counts of "unclear" or free-form focus slogans.
- Meaning on Start must come primarily from layout, iconography, rhythm, color, spacing, and tap flow.
- Start now exposes a fixed 16-area landscape.
- Area titles sit below their icon surface, not inside it.
- `Heute`-style summaries and open/stable counts do not belong on the first read anymore.

Current fixed tiles:
- `Pulse`
- `Pulse` contains a 4x4 area grid.

## Domain edit rule

When a user opens a domain from `Pulse`, the screen should stay in the same language:
- one calm info tile first
- one work tile for the actual editing below
- no stacked mini-screens for `Ziel`, `Heute`, `Quellen`, and `Muster`
- no weak helper copy like `aktiv`
- no internal labels like `Core`
- as little text as possible, with meaning carried by layout, fields, icons, and tap targets

## Core movement

The core movement of `Single` is now:

1. The user opens `Home`.
2. The day track shows fit or drift.
3. The user taps a block.
4. The overlay explains that block just enough.
5. The user makes one correction.
6. Home mirrors the result immediately.

## Interaction model

### Day track

The graph is the first read.

It should communicate:
- how full the day wants to be
- how full it actually is
- where it drifts

The graph itself stays text-light.

### Block overlay

The overlay is the real action layer.

It shows:
- block focus
- `Soll`
- `Ist`
- reasons
- current block items
- direct actions

It replaces the need for a primary `Werkbank`.

### Quick add

`+` is secondary.

It exists for:
- fingerprint and model calibration
- sources and domain setup
- research and inspection entry points

It does not carry the core product meaning.

## Navigation rules

- Home remains the stable anchor.
- Tap on a block goes deeper without feeling like app-switching.
- Back closes overlays before screen navigation.
- Share/import returns to Home.
- Optional calibration like `Lebensrad` must never block first use.

## What gives the app meaning

The app becomes meaningful when it does this well:
- makes the day readable at a glance
- helps the user correct drift quickly
- learns the user’s `Soll` locally over time
- reduces friction without forcing constant manual input

The app is not meaningful if it only looks good or stores items.

## Near-term constraints

For the next iterations:
- keep the current Home geometry
- improve meaning before adding more tools
- keep the visible surface minimal
- prefer one strong loop over breadth
- let passive signals and local learning feed Home later, not clutter it now
